User-Generated
Augmented Reality
This report is part of an undergraduate research project that investigates augmented reality
(AR) technology and whether it is possible for users (without a knowledge of AR) to create AR
content.
This report describes the design, development and implementation of a responsive webbased application that allows users to create AR content without requiring knowledge of the
technology.
This project is an extension of a project titled, Place-Based Narratives, run by the University
of Brighton and the University of Sussex, which also investigates user-generated augmented
reality. This report begins by describing the projects’ topic area and the problems the PlaceBased Narratives project faced during a previous investigation into user-generated
augmented reality. Using these problems, this report outlines the motivation and aims and
objectives of this project, setting the standard for a later evaluation.
This project has been completed in accordance with the British Computer Society (BCS) code
of conduct, and this report continues by describing how the project has met their code of
conduct. In this section, this report also discusses the ethical considerations that were
required for this project.
Following this discussion, this report explores the background research completed in this
project – an in-depth look at existing tools and technologies that allow for user-generated
augmented reality and web-based AR. Using this knowledge and the aims of this project, a
table of requirements were created and is displayed within this report. Arranged into
functional, non-functional and domain requirements, these describe the required behaviours
and functionalities necessary for the application.
This report then discusses the design, development and implementation that was required to
meet the application’s requirements and the project’s aims as a whole. The chapter titled
‘Design’ looks at the different design diagrams: use-case, data-driven model and state,
created to support the application’s back-end development, as well as user-interface sketches
that supported the front-end development. This report then discusses the development of
the application, describing the back and front-end development, and the application’s AR
development. This report then showcases the developed application with a walkthrough of
the application, shown using a storyboard of screenshots.
This report follows by describing the various forms of testing carried out on the application,
delivering the results of each test and explaining any failures or components that could not
be tested. This report continues by describing the future work required to improve the
application, before evaluating the system against the project’s aims, and reflecting on the
project and my experience of the project within the project’s conclusion.

1. Introduction
Augmented Reality (AR) is a technology that integrates virtual content within a user’s reality
[1]. Achieved using a camera and an AR developed experience, once launched or prompted
by scanning a ficidual marker, the experience launches the virtual content within the camera’s
viewport; appearing as if in the user’s physical environment.
AR is an emerging technology with increasing uses. A popular example is Pokémon Go (Figure
1), a gamified AR experience that allows users to ‘catch’ fictional Pokémon characters [2].

Figure 1 - Pokémon Go, a gamified AR experience [2]

Further examples include IKEA Place, an application that allows users to arrange virtual
furniture within their home [3], and social media platforms Snapchat and Instagram which
allow users to alter their appearance with virtual content [4] [5].
These examples show how the average user is increasingly using AR in everyday life. However,
the content within these examples is ‘fixed’; it cannot be customised by the user. The reason
being, AR development is a specialist field, requiring knowledge outside of an average
person’s general knowledge. Therefore, this project is focused on user-generated AR content;
allowing individuals to create AR content without knowing about AR.

1.1. Project Description
This project, titled User-Generated Augmented Reality (UGAR) aims to investigate how users
interact with AR and develop a system that allows a user to create AR content, without
knowing how the technology works.
UGAR is an extension of a wider project called Place-Based Narratives (PBN) [6] run by the
University of Brighton and the University of Sussex. Primarily, working with primary schools,
PBN investigates how local communities can connect with their cultural environment using
web-based AR.
In 2019, PBN ran workshops and asked primary school children to produce cardboard models
of their homes. These models produced a map of Brighton and Hove from the viewpoint of
these children. The models were digitised into 3D models and arranged onto a virtual AR map
[6].
Figure 2 shows a physical and virtual comparison of the children’s models.

The problem area of PBN is, the tool developed cannot function independently of middlemen

The motivation for UGAR is to automate PBN’s process. Automation would replace the
developers with an automated tool that would support the user in completing the digitisation
themselves.

1.3. Project Aims
UGAR should aim to produce a responsive web-based application - web-based being a
requirement for compatibility with PBN. The application should be able to capture user
content, convert captured content into a texture, create a 3D model, apply the texture to the
3D model, and integrate the textured 3D model into an AR environment. The application
should also allow for simple interaction between model and user.
Possible extensions of this project could involve identifying a long-term storage solution for
user-generated images, providing more complex interactivity between model and user, and
support the creation of more complex 3D models.
This report shall discuss and evaluate the research, design and development undertaken to
implement a solution to UGAR; a responsive web-based application that allows for usergenerated AR.

3. Requirements Analysis
3.1. Background Research
3.1.1. Existing Solutions
3.1.1.1. Place-Based Narratives Project
UGAR is part of a wider project called Place-Based Narratives (PBN) [6]. PBN is a heritage
project, working with primary school children to encourage them to explore their cultural
environment using web-based AR [6].
The children were asked to produce 3D cardboard models of their homes, which were
digitised into 3D models and arranged on an AR map, using web-based AR [6]. The problem
with PBN was the project required middlemen developers to digitise the content, which
limited the children’s involvement and experience of AR.
The process of digitisation was completed offline. Each cardboard model required the
developers to create a 3D model, take a photo of the content on each face, turn the captured
content into a texture, and apply the texture to the 3D model using modelling software,
Blender. The 3D models were saved in a glTF file format and arranged on the web-based AR
map [6]. Once launched, the experience produced a virtual map of Brighton & Hove [6].
A benefit of studying this project was the tools used to develop the experience. The project
used HTML, CSS and JavaScript to host the experience in a web browser, incorporating AR
using A-Frame and AR.js [8] [9]. Another benefit of studying PBN was it gave me the process
that UGAR needed to automate, to be successful [6].
The main issue with this project is it isn’t user-generated AR because the user did not
complete the whole experience themselves. Therefore, UGAR needs to feature automation
to be successful. Further issues discussed in this project include differing screen sizes and
their effect on the experience, the computational power of smartphone CPU and GPU units,
and a lack of functionality on iOS devices [6]. Also, results from the user testing revealed the
users felt there was a lack of instruction on how to use the tool [6].

12

3.1.1.2. Blippar Augmented Reality Butterfly
Blippar is an AR development company and their project, the AR Butterfly, uses usergenerated AR in an AR experience targeted at children [10].
The experience uses a paper template of a butterfly as the ficidual marker of the experience,
which once coloured in becomes user-generated AR. The user scans the coloured marker and
a butterfly appears, dressed in the user’s coloured texture and is animated fluttering in the
user’s reality [10]. Figure 3 shows Blippar’s AR Butterfly experience.
Blippar's experience is user-generated AR because the user completes each stage of the
experience independently.

Figure 3 - Blippar's AR Butterfly Experience [10]

A benefit of studying this project is the content generation, using a standardised approach by
colouring in a template, which makes digitisation easier. I am going to have to consider how
to make the content capture stage of the application follow a standardised approach,
especially since it will depend on the technical competence of the user.
The issue with studying this project is the lack of documentation, which would have benefited
UGAR’s development.
Blippar's butterfly is very relevant to UGAR because the experience is targeted at primary
school children and shows user-generated AR is possible, using mobile devices too [10].

3.1.1.3. Augmented Reality Colouring Books
Colouring is a technique used widely across AR experiences to create user-generated AR.
Colouring books Crayola: Colour Alive and Disney: Colour and Play require the user to colour
the pages of a colouring book, and once scanned, display a virtual model atop the paper
appearing as if in reality, dressed in the user’s coloured texture [11] [12]. Figure 4 shows both
of these colouring books.

Figure 4 – Crayola’s Colour Alive and Disney's Colour and Play AR Colouring Books [11] [12]

These books follow the same standardised approach as Blippar's butterfly; a coloured
template [10] [11] [12]. Without this template, producing a model using children’s drawings
would be extremely complex.
Having studied the development of such a book, the main challenge is generating the model’s
texture, since the coloured paper is 2D and the models are 3D. The issue is the ‘hidden’
textures that cannot be captured by colouring and have to be generated another way [13].
UGAR will have to consider this issue because the cardboard model doesn’t have content for
every face of the model.
3.1.2. Tools and Technologies
3.1.2.1. Blippar BlippBuilder
Blippar’s BlippBuilder allows users to generate AR content without needing knowledge of AR
[14]. BlippBuilder (Figure 5) allows users to drag and drop assets onto a ficidual marker to
make ‘Blipps’ that are individual experiences [14]. These assets can be animated and
customised to perform different actions, based on a user’s interaction.

Figure 5 - Blippar's BlippBuilder [14]

14

BlippBuilder can be considered to be user-generated AR because the AR content is produced
without needing knowledge of how AR works. However, it doesn’t allow the users using the
experience to create AR content; they can only interact with the content given in the
experience. Therefore, UGAR needs to allow the user to create custom content during the
experience to be considered completely user generated.
3.1.2.2. Vuforia
Vuforia is an open-source engine that supports the development of AR experiences [15].
Integrated into software such as Unity, Vuforia has an API library that allows developers to
produce their own AR experiences [15].
There are many issues with using Vuforia for UGAR. Vuforia is a platform for users to create
their own AR experiences to be deployed [15]. This doesn’t allow users using the AR
experience to create content within the experience. Furthermore, despite being open source,
the developer has to pay for their experience to be deployed, which isn’t appropriate for
UGAR [15]. Finally, the user has to have a competent knowledge of programming which
cannot be expected of a child.
3.1.3. Web-Based AR Frameworks and Libraries
With UGAR’s solution needing to be a web-based application, research was undertaken into
web-based AR technologies. There are a variety of frameworks and libraries, including AFrame [8], AR.js [9], 8th Wall [16] and Awe.media [17], and although A-Frame and AR.js are
both a requirement for PBN, they remained the preferred AR technologies for UGAR. The
reason being, many of the other frameworks were closed-source - not appropriate for UGAR
[16] [17]. A-Frame and AR.js were superior because they are designed for web-based AR and
open-source.
3.1.4. 3D Models
Another consideration was the type of model type used in UGAR because it needed to be
compatible with A-Frame and AR.js.
glTF models were used in PBN, and originally chosen for UGAR because they are more
complex and have the ability to appear more realistic [18]. However, applying dynamic
textures to glTF models was not possible, so UGAR uses A-Frame’s shapes to create the house
model. Otherwise known as primitives, the A-Frame framework has a collection of shapes,
including cubes and cylinders that were used to create the virtual model. [19] [20].

4. System Design
Using the background research and table of requirements, a system was designed to capture
and automate the digitisation of the user’s content.

4.1. Use-Case Diagram
A use-case diagram (Figure 6) was created to show the required actions users would need to
complete to use the application.

Figure 6 - Use-Case Diagram

As seen in Figure 6, the user needs to complete five actions:
‘Register Model’ - the user needs to input data about their model: the model’s configuration,
roof type and ID.
‘Capture Photos’ - the user needs to take photos of their content.
‘Upload Photos’ – the user needs to upload their photos to the project’s file-space on the web
server.
‘View Model’ – the user needs to see their model in an AR environment.

19

4.2. Data-Driven Model
Using the actions identified in the use-case diagram, a data-driven model (Figure 7) was
created to identify the pieces of data required from the user.

Figure 7 - Data-Driven Model

As shown in Figure 7, the application begins with a Setup state that collects the student’s
school, model ID and configuration of their model. The data is collected at the beginning to
determine the configuration of the virtual model and is used to create a dedicated folder and
files for each model.
The next data collected is the user’s content. As seen in Figure 7, this stage involves getting
an image from the user for each face of their cardboard model. This could be captured in the
application or uploaded, having been captured outside of the application. The data is stored
in the dedicated model folder and used to generate the texture for the model file.
Identifying the logistics of the application was necessary to identify the different stages of the
application and their interactions with each other. For example, the user’s content needs to
be stored in their dedicated model folder. This folder only exists because the data is collected
in the Setup state of the project. If the folder hadn’t been created, it would be difficult to
keep the model and its content together.

4.3. State Diagrams
4.3.1. Original State Diagram
Using the data-driven model, a state diagram (Figure 8) was created to identify the
application’s different states and the actions that occur within each state.

Figure 8 - Original State Diagram

Similar to the data-driven model, the application begins with a Setup state and upon
completion, the user is taken to the Content Capture stage of the application.
The Content Capture stage has seven states and two pathways through the application. Two
pathways were identified because the application needs to cater for models with and without
a roof.
Pathway 1: Front, Left, Right, Back, Top
Pathway 2: Front, Left, Right, Back, Back-of-Roof, Front-of-Roof
Post Content Capture, the user’s options involve viewing their model in the ‘View Model’ state
or starting again. The ability to restart was included to meet the domain requirement of
‘Allowing for repetition’.
Identifying these necessary states provided the fundamental process to produce one of the
configurations in the development stage of the project. This process was later adapted to
cater for other configurations.
4.3.2. Revised State Diagram
During user testing, two functionalities were requested to allow users to traverse backwards
between states and restart during the Content Capture stage.
Figure 9 is a revised state diagram, created to show how the application could include the
backwards traversal and restart functionality.

Figure 9 - Revised State Diagram

In this revised version, the user can traverse backwards between states: Left, Right, Back, Top,
Back-of-Roof and Front-of-Roof. The Front state is not included in this functionality, because
it follows Setup, which defines the user’s content capture workflow. Front is included (with
all other states) in the restart functionality which allows the user to restart the process again.
The diagram provided clarity in how the application would accommodate these additional
functionalities, ahead of implementing them.

Screen 4 visualises the Content Capture stage and consists of a camera viewport and buttons:
‘Take Photo’, ‘Upload Photo’ and ‘Next’. The camera viewport is used to capture the content,
and while this is a generalised sketch, each configuration would have separate pages with
differently sized viewports, constrained to match the relevant sides of the cardboard model.
A constrained camera viewport ensures a standardised approach; a key theme identified in
the background research. By constraining the camera’s aspect ratio to the sizes of the sides
of the cardboard model, the captured content is a 1:1 match. The only errors being the user
choosing the wrong model configuration or the user’s technical competence.
The buttons ‘Take Photo’ and ‘Upload Photo’, are necessary to trigger the capture and upload
the content (regardless of capture source) to the project’s file-space. Incorporating these
buttons met the functional requirements of allowing the user to capture and upload their
content inside the application.
Assuming the user has completed all states of the Content Capture stage and has content for
all sides of the model, the user is taken to Screen 5, the view model screen. By scanning the
project’s ficidual marker, the user sees their virtual model.
Designing the system with these diagrams was crucial in understanding how the application
could support the user in capturing their content.

5.1.2.1. Cuboid Creation
The cuboid is the most important shape in the user’s model because it holds the majority of
the user’s content. Therefore, the technology used to create the cuboid was important
because it impacted how UGAR would function. Two model types were considered when
creating the cuboid: glTF and <a-box>.
5.1.2.1.1. glTF with Blender
Originally, UGAR was to use glTF models because glTF was used in PBN, and background
research identified glTF to be more sophisticated [18]. I was unfamiliar with glTF, so I
experimented creating and exporting a glTF cube for AR using 3D modelling software,
Blender.
Figure 13 shows a cube created in Blender, displayed using A-Frame.

Figure 13 - A glTF cube created in Blender and exported for AR

As seen in Figure 13, it was possible to use a glTF model with A-Frame. However, it was not
possible to dynamically change the model’s texture.
Three potential solutions were attempted:
BlackThread.io
BlackThread.io is an online tool that applies textures to glTF models [25]. The tool uses a
method from three.js called GLTFExporter, which delivers and loads 3D content [26]. The tool
was unsuccessful and lacked an API.
Three.js + GLTFExporter
The three.js library was used to try and access the three.js’s definition of the texture, using
the same GLTFExporter function to swap the texture [26]. This solution was repeatedly
unsuccessful.
Blend4Web
Blend4Web is a web-framework, which supports 3D models integrated into HTML but lacked
support for glTF models [27].

25

Considerable time passed without a solution, so I met with the PBN team and it was decided
the cuboid would be created using A-Frame’s primitive, <a-box> [19].
5.1.2.1.2. A-Box
A <a-box> was used to create the cuboid of the virtual model. The decision being it is
compatible with A-Frame, and A-Frame defines model textures outside of the model's
definition, making it possible to swap textures - allowing for automation.
An attempt at texturing the cuboid was made using a typical cube map featuring all six faces
of the cube in one texture. Figure 14 shows this application and was unsuccessful because AFrame applied the texture to all sides of the cube, instead of applying the content to the
relevant face.

Figure 14 - A cube map texture applied to an <a-box> [40]

This application altered UGAR’s previous approach, whereby the user’s content would have
been stitched together using PHP to create a laid-out texture (similar to the texture shown in
Figure 14) containing all of the user’s content.
To overcome this, research was conducted into A-Frame and how it applies textures.
Currently, it is not possible with A-Frame alone to apply unique textures to individual faces of
A-Frame primitives.

match the different sides of the cuboid. The dynamic filenames allow textures to be swapped

In addition to the functional and non-functional requirements, the application was tested

UGAR failed because the application is only compatible with tablets and desktop devices.
The incompatibility with smartphones is because the constrained camera viewports are
greater than the screen size of most smartphones.
Attempts to cater for smartphones were made but smartphone cameras operate in a
portrait orientation, while tablets and desktop devices use landscape orientation, which
would have required an alternative workflow. Experimentation into catering for mobile
devices took time and focus away from producing the virtual models.
It was agreed with the PBN team, development would focus on catering for desktop and
tablet devices only because these were the types of devices more likely to be available in a
school environment.

Most of the feedback was implemented in the final version of the application. Feedback not
implemented included the visual feedback for downloading and uploading the user’s content;
not implemented because the download and upload are both actions that provide visual
feedback already. The download button downloads the user’s content to their local storage,
and the upload redirects the user to the next stage in the application.

8. Future Work
Despite producing a functioning application, further work is required to improve UGAR’s
application.

8.1. Compatibility with PBN
The application needs to ensure it is compatible with the PBN project and the AR map [6].
Ensuring compatibility has been a consideration throughout the design and development of
UGAR. However, more functionalities may be required to translate the models onto the AR
map. UGAR’s application and the whole experience should then undergo testing with children
because this was not possible during this project. The experience should also include the
application’s instruction manual, not currently implemented.

8.2. Model Inequality
The image quality in the images used to generate the models’ textures is inconsistent. Bigger
models’ viewports are resized to fit within the user’s screen. However, the scaling results in
lower texture quality because the images used to create the textures are smaller.
Investigation could begin by rotating the viewport, so the model’s content fits within the
constraints of the user’s device. However, this would also require rotating the resultant image
reversely. A similar investigation was completed in UGAR’s development, to cater for mobile
devices and shows this implementation is possible.

8.3. Aliasing
The models suffer from aliasing along the join between the prism and the cuboid objects. An
investigation should look into ways to improve the aliasing. The application already uses an
anti-aliasing renderer, but future development could research other rendering options
available in A-Frame [39].

8.4. User Interactions
The current user interaction between model and user is very simple, the user can initiate an
animation of their model rotating clockwise. A-Frame does have a functionality called lookcontrols that allows the user to drag and rotate their model 360 degrees on both the X and Y
axes [31]. However, look-controls is only compatible with devices that have a 16:9 aspect
ratio. Future development should investigate look-controls and determine whether it is
possible to create a version that works with all devices.

8.5. Update Model
Future development should create a function that allows the user to update a single face of
their model. Currently, if the user wishes to update their model, the user has to traverse
backwards between faces or restart their model. This functionality would allow the user to
update their model outside of the content creation stage, reducing the steps required to
update the model. Development of this functionality should use the current lookup
functionality provided in the application and present a menu with each face arranged in a list
of buttons. The user should choose the face to update and be able to capture new content.
UGAR aimed to include this, but time constraints made this not possible.

64

9. Conclusion
UGAR aimed to develop an application that allows users to create AR content, without
needing knowledge of AR.
The application meets this aim because the user does not come into contact with augmented
reality until they view their model. The user’s purpose is to capture their content, supplying
it to the application, who is responsible for digitising the content, applying the texture to the
model, and ultimately producing the user’s virtual model.
The identified digitisation process involves capturing the user's content, creating a texture,
applying the texture to a 3D model, and incorporating the 3D model in an AR environment.
To evaluate UGAR, the application should be tested in its ability to perform those five tasks
[6].
This evaluation should also consider how UGAR compares to the tasks completed by the
middlemen developers in the PBN project [6]. Only then will it be possible to determine
whether the application is successful.

9.1. Capturing User Content
The application is successful in capturing the user's content because it is possible for the user
to capture their content within the application. The application is successful by accessing the
user's camera and saves time for the user, removing the need to switch applications to
capture the content. Another factor in UGAR's success is accuracy. Accuracy is guaranteed
through the use of variously sized constrained camera viewports, which are a 1:1 map
between the application and cardboard model content.
The use of constrained camera viewports boasts accuracy because they are an example of
how UGAR uses a standardised approach. Errors are reduced because the only possible errors
are user errors, based on the technical competency of the user. The application compensates
for these errors by providing functionalities, such as backwards traversal and restart to
support the user in creating an accurate representation of their cardboard model content.
The ability for the user to capture their content further shows success because the PBN
project required middlemen developers to digitise the user's content [6]. UGAR is successful
because it doesn't need experts. Instead, UGAR guides the user in how to capture their
content, allowing the user to complete the action themselves.

9.2. Creating Texture
The application further shows success in creating a texture for the user's model because it is
automated with the only inputs being the user's content. The textures are captured per face:
front, left, right, back, top/roof elements and are an example of how the application uses a
standardised approach. The user must capture the content in this particular order, and the
application guides the user through this process by telling the user what content they are
capturing.

65

The workflow is successful because it ensures the correct content is captured, resulting in an
accurate virtual representation of the cardboard model. The digitisation is automated,
completing all the modifications required to prepare the textures for application - another
example of UGAR' success because PBN required the textures to be made manually by the
experts. UGAR is successful because it produces the same result, making the experts obsolete
[6].

9.3. Applying Texture
The application successfully applies the textures to the virtual model because this also
happens automatically. If the user has captured content for all sides of their model, the user's
model will appear fully textured. The application achieves this by storing the content
alongside the relevant model, naming each piece of content with a dynamic filename that
matches the relevant face on the cardboard model. This approach ensures the correct content
is applied to the correct model, and again is superior to PBN being automated and producing
the same result [6].

9.4. 3D Model and an AR Environment
The final criteria to assess UGAR is the ability to produce a 3D model within an AR
environment.
UGAR achieves this aim because UGAR can produce nine unique models within an AR
environment. UGAR uses A-Frame to host and create the models, using a ficidual marker to
launch the experience. The application has nine different model files that share the same
texture definitions, which make them a generic template that is copied and stored in the
user's dedicated folder. The model is made unique by referencing the user's content. The use
of nine templated models is another example of a standardised approach because the user
chooses the virtual model that best fits their cardboard model. Therefore, UGAR improves on
the PBN project because the application could not be automated without these templated
models as each model would need to be made individually [6].

9.5. Reflection
In addition to UGAR's performance, UGAR has also been successful in improving my technical
and soft skills.
UGAR has allowed me to apply and extend my knowledge of HTML, CSS and JavaScript.
Accessing the user's camera is an example of how my understanding of JavaScript has
developed because I had previously used JavaScript for more simple functionalities. UGAR has
developed my understanding of JavaScript, so it is a more functional technology in my
repertoire. Using HTML and CSS has allowed me to develop a greater understanding of how
they work and supplement each other.
UGAR has also introduced me to AR; a field of computer science not previously studied.
UGAR's A-Frame development was particularly challenging because AR remains a relatively
new technology, and A-Frame bears evidence of this with limited documentation and
support. However, this improved my problem-solving skills because I had to create a solution

Working on UGAR has been an enriching experience. UGAR's application is a successful
