Introduction
1.1 | Project Overview
In this project, I have both developed a Tower Defense (TD) game and a Reinforcement
Learning (RL) agent within the game. I hope by using RL, I can break the current limitation on most game Artificial Intelligence (AI) being predictable and lacking complicated
tactics. I also aim to prove that it is practical to use "non-programmatic" AI to enrich
players’ experiences.

1.2 | Motivation
Being a keen gamer since childhood, I have witnessed improvements of video games in
many different areas. From pixel arts to photo realistic rendering, from text based dialogue to stereo sound effect from the environment, game developers has made incredible
improvement in various areas – improvements in AI is also significant, however it is
community consensus that game AI is still generally speaking, dumb and predictable.
On the other hand, there is also rapid improvement in recent years in AI, in particular,
there are quite a few RL agents that consistently beat human players which were thought
to be impossible. This makes me wonder why is it the case that not many games are
utilizing AI and gaining commercial success. Hence I want to develop my own game to
see the intricacies of incorporating RL into game development. This means that I wanted
to focus on building a working game prototype, that uses RL in the game AI.
The game is a TD game, where enemies’ target is to reach the target position and the
player’s goal is to defend the target position with turrets that can damage the enemies,
this implies that our RL agent will need to perform tactical decision to approach the

This project will give me the opportunities to delve into various aspect of game

1.3 | Findings and Achievements
In this project I used RL to generate an AI opponent in a TD game I built from scratch.
While the performance of the trained agent is rather underwhelming, this is at least
in part due to the lack of computational power which is typically needed to generate
AI agents via RL in a realistic game. However we were able to investigate several
factors which affect the agents that are produced by RL. In particular, we looked at the
influence of the encoding of the input data and the curriculum(the layout of the training
environment depending on the agents performance during training) used for RL. We
saw that both aspects have important impacts on the performance and robustness on
training an RL agent.

Game Development – Overview
2.1 | Overview
Our main purpose of the project is to incorporate RL into a game, which means we have
to both develop a game and the RL agent. Given the limited timeframe, the game has to
be both simple to implement yet the AI will still need to have a basic level of complexity
such that it is challenging enough for the AI to learn – otherwise we could not prove that
it is indeed practical to incorporate RL in game development.
Base on this requirement, I have decided to create a 3D TD game with some tweaks
on some of the existing conventions. In the following sections, we will first outline what
our game will be like, followed by a requirement analysis, and then finally we will briefly
discuss what technology is used while creating the game.

2.2 | Game Description
As mentioned in section 2.1, I will be building a TD game with a few tweaks as our
platform for training our RL agent. In this section I will first briefly introduce what is
a TD game, and then I will elaborate how a few fundamental rule is changed for our
purpose.

2.2.1 | Typical Current Tower Defense Game Structure
TD has been a long-living subgenre of video games, which emerged in the early stages
of video games history and is still popular today. A typical TD game often looks like
the screenshot shown in figure 2.1 and is generally structured as follows(graphical
representation can be seen in figure 2.2):

2.2. Game Description

Figure 2.1: Screenshot from Flash Element, a classic representation of tower defense game

Figure 2.2: Typical game flow for TD game

1. Game begins, the player is given a certain amount of money
2. The Construction Phase - Player will construct turrets at different locations on the
map, various types of turret are available through upgrading from the “base turret”,
they all serve different purposes such as dealing damage or slowing down the
enemy. Depend on the specific implementations, the player has unlimited or
limited time to think how they should organize their defends for the next “wave”
3. The Wave begins - Enemies are spawned and start marching with a predefined path
to the player’s base. Every time an enemy made contact with the player’s base, a
certain amount of health of the player is reduced. The wave always consists of a
fixed number of enemies and the number increases as the game progress. Game
ends when a player loses all his health or when the last wave is finished (every
enemy of the last wave is killed), the player can still upgrade turret on the fly when
they earn enough resources through killing the enemies.
4. Go back to construction phase if there is another wave, else the game/level is
finished.

2.2.2 | Deviations
From the previous subsection, one can immediately see that there is not much room for
RL – after all, the agent just march in a predefined path. To allow our RL agent to shine, I
have made a deviation such that the enemies need to make contact with the constantly
moving player character itself, instead of a static base that cannot move. Besides, the
battlefield will be significantly more open with less obstacles, which means there are
more paths that can reach the player, with some paths potentially being significantly less
risky(with less turrets in the path, or just short enough that the turret does not matter)
than others.
The fact that numerous path is available means that the AI needs to be tactical to be
effective. For example, taking the shortest path is not necessarily the best option when
another longer path is much safer. The agent will need to learn the tactical aspect of the
game, which is a task complicated enough to be challenging for proving the practicality
of using RL in game development.

2.3 | Requirement Analysis
This section outline some of our requirement on our TD game, note this list is not
comprehensive as most of the requirement of a TD game is too subtle to be discussed in
detail. We will focus on aspects that encourages active decision making from either player
or the enemies(agent) as it is what is important for our project to prove the practicality of
utilizing RL in game development.

2.3.1 | Functional Requirements
A list of other functional requirements is as follow:
 Players should be encouraged to make active and real-time decisions during the

wave.
 Enemy should be "smart" - players are discouraged to out-gun them, players need

to out-smart them to survive longer, instead of simply fighting them with superior
firepower.
 Different strategies should be viable to the player, such as

– Turtling, where the player set up heavy defense in one location and very rarely
get out of the fort.

– Roaming, where the player runs around the map and seek opportunities to
strike at the target and doesn’t rely on heavy defenses.
– Spawn Camping, where the player set up heavy defense positions on the
enemy spawn point, such that any enemy spawned at that point will be killed
instantly.

2.3.2 | Non-Functional Requirements
The following is a list of non-functional requirements, that can also be viewed as a "design
guideline":
 In contrast to most existing TD, the target of the enemies must be mobile (controlled

by player) during attack phase in a relatively open terrain.
 Turret can be placed by the player within a certain range.
 Player has some self-defending power - walking alone in the wild should be

dangerous but not impossible.
 Enemy are human-like and will make decisions depending on their surroundings.

Tactics such as flanking and baiting (where some objectives for the player pretends
to be an easy target and springs a surprise attack when the player approaches)
should be deployed by AI when necessary.
 3D view with no global map available, allowing enemies to exploit players’ limited

vision.

Game Development – Background
3.1 | Introduction
As our goal is to prove that RL can be practically utilized in game development, even if
our final product is merely a prototype, it needs to possess a certain level of complexity
otherwise it could not function as a proof that RL can be practically used in commercial
game development.
To create a slightly more polished prototype within the limited timeframe, I have
used Unity throughout the entire development process as it provides various utilities
that shortens the development cycle and allows me to put more focus on the RL. As such
we will give a brief introduction on Unity in the upcoming sections which is important
to our discussion in chapter 4.
Besides, to expose myself to various areas of game development. I have created and
animated our enemy agent’s 3D models from scratch through procedural animation. We
will briefly introduce procedural animation using Inverse Kinematics (IK) through gradient descent, as it is one of the more complicated system that I built and has introduced
problems that we were not able to resolve within the timeframe. The more detailed
implementation and the problem we faced will be further discussed in chapter 5.

3.3.1 | Inverse Kinematics
When working with procedural animation, there is often the need to put a certain part
of the animated objects into a specific position. For example, if we want to animate a
character picking up a mug, we need the character’s hand in the mug’s handle’s position.
In the process, the animated object must also adhere to certain constraints – using the
same example, the joints of the character need to be in a natural angle that can be achieved
by a human. This seemingly simple requirement is, in fact, a very difficult task to solve.
So much so game developers resort to robotics technology – IK.

Figure 3.2: A typical linked chain, where position of the end effector J2 is calculated by
FK
In robotics, a linked chain constructed with joints J0 ...Jn and links between joints
L0 ...Ln−1 . Given a linked chain, angles of all joints and lengths of all links, FK refers to
the use of kinematics equation to find out the position of the last joint Jn (more often
known as the end effector). Figure 3.2 shows a typical linked chain and how FK find the
position of the end effector.

Figure 3.3: Given the same linked chain and the same target effector position J20 , how
multiple solution can be yielded with IK, note although rotated, length of both L1 and L2
remains the same.
Intuitively, IK is the mathematical process that given a linked chain and the required
position of the end effector, calculate the angle of all joints such that the end effector
will be in the position of the required position(we refer the solution as s), this reversed
operation yields the name of "Inverse" kinematics. One thing to note is that unlike FK,
IK very often has multiple valid answer, and it is up to the implementation to choose the
best solution. Figure 3.3 demonstrates how multiple solution can be yielded by an IK
solver given the same input.
Typical algorithm for solving IK includes gradient descent and Forward And Backward Reaching Inverse Kinematics (FABRIK) introduced by Aristidou and Lasenby
(2011). I chose to create a gradient descent IK solver because of its relative ease to implement and tuning, however we will discuss about how FABRIK can potentially improves
our game in chapter 8.

3.3.2 | Inverse Kinematics through Gradient Descent
Gradient descent is an iterative optimization algorithm to find a local minimum of a
function f ( x ) (Ruder, 2016a). When used as an IK solver, f is a function that given an
input x where x is the rotation of all joints j0 ...jn−1 , output the distance between jn the
end effector and the target position t.
This is a simple implementation but not without caveats, one can immediately see
how the local optimum might not be the global optimum – however, this is sometimes

3.3. Procedural Animation

desired in an IK solver. This is because even if there is some other optimal solution s
available that results in a solution that does not, if the configuration in s is significantly
different from the current configuration, it will imply that the linked chain will need to
make a huge movement to apply solution s. This huge movement is often not desired as
it breaks the smooth interpolation in close vicinity we require from the IK solver, which
is needed for animation purposes.
One advantage of using gradient descent is its relative ease to tune, when using
an IK solver we often want to add some additional constraints, for example, it is often
desired that the linked chain to use a more "natural" configuration, this naturalness is
something that is harder to accomplish with other IK algorithms, but can be relatively
easily achieved when using gradient descent through adding a comfort function into the
function f , implying when performing gradient descent the algorithm will also take the
naturalness into account(Baerlocher, 2001).
The disadvantage of using gradient descent is that it can be relatively slow if one

4.2 | Challenges – Inversion of Control
As a final note on the environment, I want to briefly mention an ongoing problem
during my development when trying to achieve IoC - an important principle in software
engineering.
IoC encourages an "inverted" flow of control as compared to traditional control flow
in design. A real world example will be a desktop in contrast of a laptop: when you use
a desktop, the desktop controls where you sit(as it is stationary), you are "controlled" by
the desktop; in contrast, with a laptop, you "control" where you use it. IoC encourages
in-dependency between modules, and freedom to replace modules with little to no
side-effects.
In software engineering, this means that the software/object consumer gains more
control over the software/object itself. Thus it also sometimes known as the "Hollywood
Principle" - Don’t call us, we call you.
There is many different ways to achieve this, one of the most popular being Dependency
Injection (DI), which refers to a pattern that "any dependency of an object should be
provided by the constructor or the caller". Doing so improves flexibility, maintainability
and testability.

6.4.1 | Long Term Credit Assignment
RL’s biggest weakness is perhaps the fact that it is difficult for the algorithm to determine
which crucial action or series of action will lead to the reward. To counter this problem,
one approach is to assign credit frequently, this is known as shaping rewards as it shape
the behaviour of the agent.
However, if we are not careful the agent might learn to exploit our shaping reward
- instead of doing the crucial action that leads to the final reward, it might repetitively
do a series of action that leads to a huge amount of shaping reward that eventually out
weight the final reward (ClarkDario and Amodei, 2016). This is known as reward hacking.

6.4.2 | Exploration and Exploitation
RL’s core is to stochastically search through the action space given a state. Rewards will
be given when the action is beneficial to the agent, and sometimes punishment will be
given to detrimental action.

However, what seems to be correct now might not be actually the optimal behaviour,
and might lead to huge loss after a period of time, and vice versa. To counter this problem,
curiosity heuristic is introduced to encourage agents to explore the world when there is
an unknown situation. This sometimes solves the problem very well, but also introduces
its own problem, such as the agent procrastinating on taking any action because it does
not know what to anticipate in the current situation for an extended period. This is also
known as the TV-problem as, it was once reported that the agent got addicted to TV
when exploring a maze (Savinov et al., 2018).

6.5. Reinforcement Learning Algorithms

In general, a typical RL training algorithms will be similar to figure 6.2. The agent
will be put into an environment, given an input that represents the current state of the
environment and the agent will then be required to output an action. The action will be
rewarded by a certain "reward scheme".
The goal of the agent is to learn a policy(often notated as π), that will maximize
the reward that the agent get from the reward scheme. There are two families ofRL
algorithms that aim to tackle this problem - value-based and policy-based algorithm.

Figure 6.2: An overview of training process(feedback loop) in RL

6.5.1 | Value-Based Algorithms
To find the optimal policy π, value-based algorithms first find out "how valuable is it to be
to in a state s". Which means before learning the optimal policy π, value-based algorithms
first seek the optimal value function V (s) by exploring the environment through acting
randomly and sample the values of different state-action pairs. Quite naturally the value
is often directly related to the reward one can get to be in a certain state. The aim of
value-based algorithm is that it is relatively trivial to find the optimal policy given the
optimal value function.
A typical example of value-based algorithm is Q-Learning (Sutton and Barto, 2018),
where the value function also takes the action a as input to calculate the Q-Value which
estimates the maximum cumulative future reward for taking an action in a certain state.
Value-based algorithms tends to be sample efficient as the agent can learn from any
previous trajectory sampled from the same environment. However they often suffer
from curse of dimensionality – the various problems one encounter when organizing and
analyzing data in high dimensional-spaces. This is especially true when the input/output
space is continuous, and hence would be computationally expensive to learn the correct
value function through exploring the continuous space, such as the environment of our
TD game.

6.5.2 | Policy-Based Algorithms
Policy-based algorithms try to find the optimal policy π without the aid of a value function.
That is, instead of relying on the exploration of the environment and approximating the
optimal value function, the agent will make actions according to a certain policy for the
entire episode1 of training, and only receive a cumulative reward after the episode is
finished. The policy will only then be updated if it is strictly better then the previous
policy, and hence the name - policy-based.
Most policy-based algorithms utilizes gradient descent and hence also suffers from
the disadvantages associated with it, such as the risk of over-stepping the global optima
due to a large learning rate or getting stuck in a local optima when the function to be
optimized is non-convex. A typical example of policy-based algorithm is REINFORCE
published by Williams (1992).
Policy-based algorithms are less sample efficient as, unlike value-based approaches,
policy-based algorithm are not able to learn from the past. However policy-based algorithms are often computationally simpler, especially in a highly stochastic environment or
a continuous space, where approximating a value function is computationally expensive.

6.5.3 | Actor Critics
When there are 2 schools of algorithms, each with their own merits, one natural strategy
is to combine their advantages, and hence actor-critic is introduced. In actor critics
methodology, the agent is trained in two parts:
Actor which strives to learn the optimal policy π and is responsible to make actions
Critic which evaluates the action taken by the actor by learning value function, and
suggest how the policy should be updated.
The general architecture of actor-critics methodology can be seen in figure 6.3. Where
we can see that after an agent make an action based on a policy, the environment will
respond to it and produce a reward based on the reward function, the reward is then feed
into the critics(the value function), and gives reward through a TD(Temporal difference)
error function – TD error describes the expected reward changes by moving into a new
state with the action taken by the agent in the current timestep. After the reward process,
the agent will again receive the input state and output the action. This process continues
and the agent eventually learn the estimated optimal policy of the environment.

The perhaps most important improvement on actor-critics is the Advantage Actor
Critic (A2C) model, where instead of using a value function V (s) to aid training, we use
an advantage function, defined as A(s, a), that output "the advantage of taking an action
over the other action on average", hence the name "advantage" function.

6.5.4 | Algorithm that I Used – Proximal Policy Optimization
PPO is in a nutshell, a variant of A2C released by OpenAI that aim to achieve high and
reliable performance with low computational complexity, with as little hyper-parameter
tuning as possible(Schulman et al., 2017).
We will be using PPO throughout the entire development process. As PPO typically requires less hyper-parameter tuning, which requires a huge amount of time and
computational power which we do not have.
As previously mentioned in section 6.5.2, most policy-based algorithms depends
on an optimization method known as gradient descent, which is either too fast and can
easily falls in to local minima with a large learning rate, or painfully slow to train with
a low learning rate(Ruder, 2016b). PPO takes inspiration from Trust Region Policy
Optimization (TRPO) and attempts to prevent the agent from making drastic changes to
the policy. PPO achieve this by first measuring the difference between the old and new
policy based on their KL divergence, and clipping the advantage function for any new
policy that is significantly different from our previous policy.

6.5. Reinforcement Learning Algorithms

Clipping the advantage function is a rather soft constraint – the agent merely has
no preference for changing to a more different policy over the other, the agent is not
forbidden from choosing a very different policy, as such the agent will inevitably make
bad decision once in a while, yet PPO successfully strikes the balance between simplicity
and effectiveness, and out-performs other state of arts alternatives in many environments(Schulman et al., 2017).

6.5.5 | Curriculum Training
During our development of RL agent, curriculum training is one of the technique that
we used extensively, and has gone through couple of changes throughout the course of
development.
Curriculum training is a technique that based on the believes that some task is too
difficult for the agent to learn from scratch, and it is easier to learn the task if we simplify
it at first to guide the agent into the right direction. After that we gradually increase the
difficulty of the task until it is identical to our ultimate goal. We believe that it will be
easier for the agent to converge to the optimal policy through this process of learning a
slightly harder problem every level. To put it simply, if we want the agent to run, we first
teach the agent how to walk.
In RL, this is achieved by changing the environment according to the agent performance. When the agent perform consistently well enough, it will be progressed to a new
"level", the higher the level is, the harder the task will be, and more similar the scenario
will be with the actual gameplay. Curriculum training is commonly used in complicated
RL task (Gupta et al., 2017).

8.4.1 | Generative Adversarial Imitation Learning
GAIL is a type of imitation learning inspired by Generative adversarial network (GAN).
In GAIL, instead of only training one agent that makes decision, GAIL also trains another
model, known as the discriminator. The output of discriminator indicates how similar the
behaviour of agent is to the experts demonstration. Due to limitation on word count, I
have provided a summary over imitation learning in appendix 8.4.2.2, and will refrain
our following discussion to GAIL specifically.
In GAIL, after the agent performs an action, it will receive a reward from the discriminator depending on the similarity between the action taken by agent and the action
taken by experts – the more similar it is, the higher the reward. Thus the agent will try to
mimic what the expert does in order to maximize potential reward.

60

Chapter 8. Results & Discussion & Future Work

8.4. Future work – Improvement on Agents

However, the longer the discriminator is trained, the better the discriminator gets
at distinguishing actions taken by agent from actions taken by experts. Intuitively, it
becomes harder and harder for the agent to "fool" the discriminator, and hence the agent
will also need to get better at mimicking the action of experts. The competition between
the discriminator and the agent leads to the name of "adversarial" imitation learning.

Figure 8.1: Demonstration on how GAIL works
GAIL aims to solve the problem in Behavioral Cloning (BC) where a large amount
of demonstrations is required and the risk of compounding error leading to undesired
behaviour, whilst avoiding the huge computational cost that roots from the principle of
Inverse Reinforcement Learning (IRL)(Ho and Ermon, 2016).
One may note that it is very similar to our original RL framework – the agent take as
input a state, output an action and is rewarded for that decision, with the ultimate goal
of agent learning a policy that maximizes the reward.
Naturally, one can combine GAIL with other source of reward, in our case, our
original extrinsic reward function. After combining with the original extrinsic reward
function, the original model from figure 8.1 evolves to figure 8.2.
The ratio between the GAIL reward and the extrinsic reward can be tuned by a
hyper parameter. The more we value the GAIL reward, the more important for the
agent to mimic the behaviour of the expert, and the quicker in theory it should converge.
However it also means that it is less likely to emerge new behaviour which we mentioned
in section 6.4.3, which is one of the advantage of using RL to develop the AI.

tasks. Their testing demonstrated remarkable results and out-performed many other RL
techniques.
Given the high similarity between their test and the nature of our game, their experience show us that it is possible that the task is actually dividable, yet we failed to divide
it effectively.
One quite natural advantage of this approach is that it is possible to switch any of the
sub-agent to use approaches mentioned in appendix 8.4.2.2. This can be very useful if
any of the sub-tasks is easily programmable, we can then focus on training one agent at a
time, making training significantly easier.

is particularly useful in RTS games where units can perform "soft-retreating" (backing

danger or to rout from battlefield to regroup and fight another day.

which in most case is all we need - after all as long as the player think the AI is intelligent,
it does not really matter if it really is.
However, it is worth noting that influence maps are merely a data type - they only
helps to encode data of the battlefield into a form that can be easily interpreted by AI,

Tactic and Planning
In a simple strategy game, even random tactics employed by AI might be perceived as
intentional by human players (Wetzel, 2014). This is due to the fact that human tends to
rationalize any observation on their surroundings.
However, random tactics will soon look non-sensible as the game’s complexity
increases. For this reason, a lot of effort has been invested to help AI make better
decisions. It is still somewhat manageable when the game’s information is complete, such
as chess and Tic-Tac-Toe, yet once the information is incomplete (i.e, not all information
is available to all players), it quickly gets out of control.
Most common implementations nowadays relies on an intensive amount of maths
and/or custom tuning by designers. We will discuss some of the implementation below.

Finite-State Machine
As mentioned by Cerpa (2014), Finite-State Machines (FSMs), such as example shown
in 2, are widely used for programming behaviour and planning systems. They are easy
to implement and can be very effective, especially for less complicated games. Moreover,
it forces the programmer to think of every edge case, which although prevents emergent
behaviour, reduce bugs in production environments. Besides the AI can be relatively easy
to customize as the game develops, from change of general behaviour to a performing
specific action.

Figure 2: Simple FSM created by Barrera et al. (2015)
However, FSMs do not scale very well and can easily get unmanageable in large
and complex projects. Besides as there is no emergent behaviour, the player will find
it repetitive and can easily find patterns after multiple playthroughs, thus massively
reduces re-playability.

Goal-Oriented Action Planning
Goal Oriented Action Planning (GOAP) is an abstraction that helps AI to plan a series
of actions to achieve a desired state. Each action can have both preconditions and effects,
by chaining a series of effects to accomplish different preconditions, the agent will
73

Tactic and Planning

eventually find an action that can be performed and has the desired effects to change the
environment such that it satisfies a desired state.
In practice, it is often used for complicated AI and has a decent result. It is also much
more scalable as the programmer does not need to manage a huge FSM. It is often used
to create AI that would be too complicated for FSM to handle.
However, it again requires a designer or gameplay programmer to create every
possible action and specify their relative cost - which again, requires a lot of effort and
fine-tuning. Besides, the biggest weakness of GOAP is perhaps someone will need to
assign a goal for it such that it can search for the best series of action to be taken in order
to satisfy the goal, which brings us back to our initial question: how do we find out what
is our goal?
To counteract the problem of goal finding, Cerpa (2014) has suggested an approach
where goals are driven by motivation - a high-level process that detects needs and creates
goals to be accomplished.
A similar approach is also employed in Civilization V (Butts, 2010) where the army’s
movement is separated into a top-down hierarchies - Grand Strategic AI decides how to
win a game, Strategic AI manage empires and city as a whole, Operational AI decides
which battle to fight and allocate necessary forces, and Tactical AI that controls units in a
local scale. In practice, this extra layer often produce good enough results.

Simulation-Based Planning
In contrast to the above more mathematical approach, Sailer (2014) has suggested an
empirical, simulation-based approach for RTS. They demonstrated that it can be run
efficiently through some optimization, and can adapt to human player’s frequently
changing tactics.
Simulation-Based Planning functions by simulating possible outcomes after the AI
employs a certain tactic, and maximizing the payoffs. It also uses opponent modeling to
exploit players’ weaknesses. To avoid the search space being enormous, it restricts the
AI choice to a set of predefined policies such that the program could be run efficiently in
real-time.
Yet once again, it requires the designer or the AI programmer to create every possible
policy in advance, requiring massive resources to be invested into fine-tuning. Besides it
is not known whether it is suitable for more complicated games - after all Sailer (2014)
has only tested this approach on a very simple RTS.

Imitation Learning
Imitation learning is based on the idea that sometimes it is intrinsically difficult for
the agent to discover the optimal policy. So instead of throwing the agent into the
environment without any guidance, we provide the agent with some demonstration to
show how the task should be done.
There are various ways to achieve imitation learning, we will briefly discuss two of
which that is widely used, and discuss GAIL which is used in our development in a later
section.

Behavioral Cloning
BC is one of the simplest implementation of imitation learning. Proposed by Bain and
Sammut (1999), it has been proven that BC can learn tasks very quickly with reasonable
performance. BC is a supervised learning technique that heavily depends on human
demonstrations. In BC, the agent is trained to learn a policy that mimics the policy
demonstrated by the expert(often a human), which is particularly useful for tasks which
are hard to capture by a reward function but can be easily demonstrated. Some typical
example including flying a helicopter or driving a car.
However, the downside of BC is that if the agent is put into a state that is never
encountered by the experts, there is no basis for the agent to learn or reference from.
This can happen when small errors adds up and leading the agent to a state that the
expert has never visited before(Ho and Ermon, 2016). As a result BC often needs a large
amount of data for it to learn effectively, which can be hard and costly to come by in
some situation. One way to make up for the enormous amount of data required is to
use data augmentation (van Dyk and Meng, 2001), which is a widely used technique
in machine learning to help get enough data for the algorithm to function properly.

Inverse Reinforcement Learning
IRL is fundamentally different from BC - instead of learning the policy from the expert,
IRL attempts to learn the reward function behinds the expert’s policy.
The reasoning behind this is that it can be difficult to model the reward function that
guide to a correct behaviour(Arora and Doshi, 2018), so instead of modeling the reward
function manually, we first learn the reward function of the expert - or in other words,
we learn the intent of the expert. Only after that do we continue with our usual training.
Note that after IRL it does not matter what algorithm we used to optimize the policy.
The fact that we did not define the reward function, but rather learn the reward function
from the experts, gain the name of inversed RL.
The idea of IRL is elegant, yet the problem lies with the immense computational cost
as IRL essentially requires solving a forward problem(finding the optimal policy) in an
inner loop of finding the optimal cost(reward) function(Ho and Ermon, 2016). Besides,
we should also note that IRL does not directly yield the policy, and learning the policy
even when given the correct reward function may not be trivial.
